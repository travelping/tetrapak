Vision

Make it easy to package Erlang/OTP applications

  • This was the original motivation and it's still the main selling point.
  • No other build system provided enough facilities to implement this in 2010

Provide UNIX command-line integration for Erlang/OTP tools

  • Many Erlang tools (EDoc, Xref, ...) are only available through Erlang APIs.
  • Tetrapak should make using those tools a straightforward task.

Allow make-style extensibility and convenience in a pure-Erlang environment

  • Erlang projects should be able to extend their build system using
    programs written in Erlang.
  • Tetrapak should provide convenience APIs to make this easier.

5/11

Architecture (Launcher)

  • Launcher is implemented as wrapper around erl
  • Argument parsing is done by OTP init, so application config etc. can be set
  • Special I/O server is used instead of user_drv because we might start a shell later
  • CLI starts the context process and runs the given tasks
  • Exit status is set according to whether a task has failed

6/11

Architecture (Task Execution)

  • The tetrapak:boot task is always executed first
  • The boot task loads the config and compiles local tasks
  • Every task runs in its own process
  • Every task has its own I/O server
  • Output is grouped by task so it's easier to read, even though they execute concurrently
  • Tasks may require other tasks dynamically, dependencies are resolved at runtime
  • Tasks may return a key-value environment
  • The return values of a task may be accessed using tetrapak:get/1
  • If the task whose return values are read has not been run, it is scheduled
  • The calling task blocks until all of its dependencies have been executed
  • Task failures cascade through dependencies
